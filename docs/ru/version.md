# Версионирование

## Формат версии

```
MAJOR.MINOR.YYYYMMDD.BUILD
```

Пример:

```
1.0.20251003.0
```

### Поля

* **MAJOR** — крупная «эпоха» проекта. Меняется при полной смене архитектуры или несовместимых изменениях API.
* **MINOR** — веха развития. Меняется при добавлении нового модуля или больших совместимых возможностей.
* **YYYYMMDD** — дата релиза в формате `ГГГГММДД`.
* **BUILD** — порядковый номер сборки за день (начинается с 0).

## Правила изменения MAJOR и MINOR

### MAJOR

Повышается, если:

* Внесены существенные изменения в архитектура ядра.
* Внесены несовместимые изменения в API.
* Произошёл переход на другие базовые технологии (например, замена СУБД).

Пример: `1.0.20251003.0 → 2.0.20251101.0`

### MINOR

Повышается, если:

* Добавлен новый функционал.
* Введены новые возможности без ломающей совместимости.

Пример: `1.0.20251003.0 → 1.1.20251015.0`

### Только дата/билд

Не трогаем `MAJOR.MINOR`, если:

* Исправлены баги.
* Мелкие улучшения и оптимизация.
* Документация и рефакторинг.

Пример: `1.0.20251003.0 → 1.0.20251003.1`

## Использование с Composer

### Подключение по ветке

```json
"scaleum/framework": "^1.0"
```

— Подхватит все версии начиная с `1.0.x`.

### Подключение по дате

```json
"scaleum/framework": "1.0.20251003.*"
```

— Подхватит только сборки от 3 октября 2025.

### Ограничение по диапазону дат

```json
"scaleum/framework": ">=1.0.20250901.0 <1.0.20251101.0"
```

— Все релизы сентября–октября 2025.
